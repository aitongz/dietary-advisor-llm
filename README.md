# CraveAlign

CraveAlign is a research prototype for evaluating LLM dietary advisors in a craving-aware recipe adaptation setting.

The central question is not just whether a model can generate a healthy recipe. It is whether the model can adapt a food someone actually craves while respecting patient-specific constraints such as allergies, hypertension-related sodium reduction, or type 2 diabetes-related sugar reduction.

## Research Idea

Current meal planners often optimize for generic healthiness. In real life, users have cravings, cultural preferences, limited time, and medical or dietary restrictions. A useful dietary advisor should be evaluated on whether it can balance:

- Safety: no allergens or dangerous recommendations.
- Constraint adherence: follows the user's stated dietary restrictions.
- Feasibility: uses realistic ingredients and cooking steps.
- Nutrition improvement: makes the dish meaningfully healthier.
- Preference fit: reflects cultural and individual taste preferences.
- Trade-off quality: responds to requested strict, balanced, or gentle adaptation.
- Craving preservation: still feels like the food the user wanted.

This framing connects to human-in-the-loop evaluation because several of these dimensions are hard to judge with automatic metrics alone. Annotator disagreement is expected and should be treated as a signal for improving rubrics, guidelines, and model behavior.

The longer-term version would also collect post-recommendation feedback: whether the user actually made the recipe, whether it satisfied the craving, and which substitutions felt unrealistic. Craving analysis can include sensory features such as crunch, creaminess, spice, temperature, and smell, as well as cautious nutrition-related hypotheses such as whether the user may need more protein, iron, or energy. These hypotheses should not be treated as medical diagnoses.

## Run the Prototype

No API key is required.

```bash
python3 src/evaluate.py \
  --cases data/sample_cases.jsonl \
  --outputs data/sample_model_outputs.jsonl \
  --out results/pilot_metrics.csv
```

The script compares sample baseline outputs against validator-guided outputs and writes a CSV with per-case and aggregate metrics.

## Data Note
Please note that the current prototype uses **synthetic data** (`sample_cases.jsonl` and `sample_model_outputs.jsonl`) to demonstrate the end-to-end evaluation pipeline without requiring real user medical/dietary data. In a production scenario, this synthetic benchmark would be replaced by diverse, human-annotated datasets collected via the Anote platform.
Moreover, the synthetic baseline outputs are intentionally designed to include edge cases (e.g., hallucinating an allergen or failing a constraint) to test if the evaluation pipeline can correctly catch safety violations and score them accurately.

## Evaluation Metrics

The prototype includes automatic checks for:

- Allergen violation
- Restricted ingredient terms
- Nutrition improvement signals
- Craving preservation anchors
- Strictness calibration
- Preference and cultural fit
- Output schema completeness
- Overall weighted score

These are intentionally lightweight. The full research direction would combine these signals with expert or trained annotator review.

## Next Steps

- Expand from 10 cases to 50-100 cases.
- Add real LLM outputs from multiple prompting strategies.
- Add human annotation guidelines and measure inter-annotator agreement.
- Add strictness levels and preference profiles to each benchmark case.
- Add post-recommendation user feedback as an evaluation signal.
- Use a nutrition database for better nutrient estimates.
- Add clinician or registered dietitian review for high-risk profiles.
